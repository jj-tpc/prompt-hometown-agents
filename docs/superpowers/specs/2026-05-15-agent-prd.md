# Agent System PRD
**Date:** 2026-05-15
**Project:** prompt-hometown-agents
**Stack:** Next.js, OpenAI API, LangChain.js

---

## 1. 목적

NPC가 유저의 말을 듣고, 자신의 성격·습관·히스토리를 기반으로 반응하는 AI Agent 시스템. 단순 대화와 구체적 요청을 구분하여 처리하며, 모든 상호작용은 NPC의 기억으로 누적된다.

---

## 2. 핵심 요구사항

| # | 요구사항 |
|---|----------|
| R1 | NPC는 유저의 말을 듣고 성격에 맞게 반응한다 |
| R2 | 유저의 구체적인 요청/부탁은 별도 검증 파이프라인을 거친다 |
| R3 | 검증은 3단계: 유효성 → 성격 충돌 → 최종 결정 |
| R4 | 각 단계는 독립적인 OpenAI API 호출(프롬프트)로 처리한다 |
| R5 | 모든 대화와 요청 결과는 NPC 히스토리로 저장된다 |
| R6 | 히스토리는 이후 판단에 컨텍스트로 반영된다 |

---

## 3. 엔드포인트

`POST /api/agent/interact`

**Request:**
```ts
{
  npcProfile: NPCProfile      // NPC 고정 특성 (성격, 습관, 말투 등)
  npcMemory: NPCMemory        // NPC 히스토리 (LocalStorage에서 전달)
  userMessage: string         // 유저 입력
  gameState: GameState        // 현재 게임 상태 (시간, 아이템 등)
}
```

**Response:**
```ts
{
  responseText: string        // NPC 응답 텍스트
  decision?: 'ok' | 'not_ok' // 요청이었을 경우에만 포함
  action?: NPCAction          // decision: 'ok'일 때만 포함 — 게임 루프에 전달할 액션
  memoryUpdate: ConversationEntry  // LocalStorage에 추가할 항목
}

type NPCAction =
  | { type: 'give_item'; itemId: string; quantity: number }
  | { type: 'move_to'; targetNpcId: string }
```

## 4. Agent 구조

LangChain.js Agent (OpenAI function calling 기반)에 `validate_request` 툴을 장착한다.

```
유저 메시지 수신
      │
      ▼
LangChain Agent
  ├── 일반 대화/질문으로 판단
  │     └── OpenAI API 1회 호출 → NPC 성격에 맞는 응답 생성
  │
  └── 구체적 요청/부탁으로 판단
        └── validate_request 툴 호출
              └── 3-Step Pipeline 실행
```

### validate_request 툴 명세

```
name: validate_request
description: "유저가 NPC에게 구체적인 행동을 요청하거나 부탁할 때 호출한다.
              단순 대화, 질문, 감정 표현에는 호출하지 않는다.
              예시 트리거: '~해줘', '~해줄 수 있어?', '~부탁해'"
```

## 5. Request 검증 파이프라인 (3-Step)

각 Step은 독립적인 OpenAI API 호출. 앞 Step이 실패하면 이후 Step은 건너뜀.

### Step 1 — ValidateChain
**목적:** 요청이 게임 세계에서 물리적으로 가능한가?

```
System Prompt:
  - 게임 세계 규칙 (존재하는 장소, 아이템, 행동 목록)
  - 현재 게임 상태 (시간, NPC 위치, 보유 아이템 등)

User Prompt:
  - 유저의 요청 내용

Output: { valid: boolean, reason: string }
```

실패 예시: 존재하지 않는 장소로 가달라는 요청, 없는 아이템을 가져다 달라는 요청

### Step 2 — PersonalityChain
**목적:** 요청이 NPC의 성격·습관·히스토리와 충돌하는가?

```
System Prompt:
  - NPC 성격 및 특성 (personality, dislikeds, habits)
  - NPC 대화 히스토리 (최근 N개)
  - 관계도 (relationshipScore)

User Prompt:
  - 유저의 요청 내용

Output: { compatible: boolean, reason: string }
```

실패 예시: 겁쟁이 NPC에게 위험한 곳을 가달라는 요청, 이전에 거절한 동일 요청 재시도

### Step 3 — DecisionChain
**목적:** Step 1·2 결과를 바탕으로 NPC 말투로 최종 응답 생성, `ok`일 경우 게임 액션도 결정

```
System Prompt:
  - NPC 말투 스타일 (speechStyle)
  - NPC 성격 요약
  - Step 1, Step 2 판단 결과 및 이유
  - 수행 가능한 액션 목록 (give_item, move_to)

User Prompt:
  - 유저의 요청 내용

Output: {
  decision: 'ok' | 'not_ok',
  responseText: string,
  action?: NPCAction  // decision: 'ok'일 때만
}
```

**액션 타입 정의:**
```ts
type NPCAction =
  | { type: 'give_item'; itemId: string; quantity: number }
  | { type: 'move_to'; targetNpcId: string }
```

`action`이 포함된 응답은 게임 루프의 액션 큐(ActionQueue)로 전달되어 실행됨.

## 6. 히스토리 관리

### 저장 형식
```ts
type ConversationEntry = {
  timestamp: number           // 게임 내 시간 (분 단위)
  speaker: 'user' | 'npc'
  message: string
  type: 'chat' | 'request'
  decision?: 'ok' | 'not_ok' // request일 때만
}
```

### LocalStorage 키 규칙
```
game:npc:{npcId}:memory
```
- `game:` prefix 네임스페이스 — 다른 앱 데이터에 영향 없음
- Settings Panel 리셋 시 `game:` prefix 키만 삭제

### 컨텍스트 전달 전략
- API 호출 시 히스토리 최근 N개만 전달 (토큰 절약)
- N은 서버 환경변수로 조정 가능 (`HISTORY_CONTEXT_SIZE`, 기본값 10)

## 7. 게임 루프와의 관계

### 대화 중 게임 루프
- 유저가 대화창을 열고 NPC와 상호작용하는 동안 **게임 루프는 계속 실행**됨
- NPC 이동, GameClock, 다른 NPC의 행동 모두 그대로 진행

### 습관/스케줄 충돌 처리
- 대화 중 NPC의 습관 시간(gameHour)이 도래하면 즉시 중단하지 않음
- 유저와의 상호작용이 완전히 끝난 후(응답 수신 완료) **행동 큐(ActionQueue)에 습관 행동을 추가**
- NPC는 대화 종료 시점부터 해당 습관 행동을 이어서 수행

### ActionQueue
```ts
type QueuedAction =
  | { source: 'request'; action: NPCAction }   // 유저 요청으로 발생
  | { source: 'habit'; habitAction: string }   // 스케줄로 발생
```
- NPC당 하나의 큐를 가짐
- 큐는 FIFO로 처리. 앞 행동이 완료되면 다음 행동 시작

## 8. 성공 기준

| 기준 | 내용 |
|------|------|
| 판단 정확도 | 명백한 요청/대화 구분이 올바르게 동작 |
| 성격 일관성 | 동일 NPC가 유사한 요청에 일관된 반응 |
| 히스토리 반영 | 이전 대화 내용이 이후 판단에 영향을 줌 |
| 격리 | 리셋이 `game:` 키만 삭제하고 타 데이터 보존 |
